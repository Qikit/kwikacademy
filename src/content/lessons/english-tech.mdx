## 1. Зачем разработчику свой пласт английского

Уровня A2 хватает на отпуск и кафе. На работе он рассыпается: разработка живёт на узком, плотном словаре, которого нет в общих курсах. На standup нужно за тридцать секунд сказать «вчера дочинил баг в авторизации, сегодня беру тикет про кэш, заблокирован на ревью PR». На code review — оставить комментарий, который не звучит грубо. В чате — попросить кого-то посмотреть пулл-реквест и не написать «make me a code review», от которого сворачиваются уши.

Беда в том, что общая лексика тут не выручает. `Make` вместо `do`, `actual` вместо `current`, `deploy on prod` вместо `deploy to prod`, `pull request` вместо `git pull` в смысле «обнови репозиторий» — каждое такое слово сразу выдаёт, что человек переводит дословно. Локально это понятно, на работе — мешает: коллеги переспрашивают, ревью буксует, оценки задач читаются криво.

Этот урок — про базовый словарь и набор готовых фраз, без которых рабочая коммуникация в IT не складывается. Не про продвинутую техническую дискуссию о distributed systems, а про каркас: процесс, код и git, архитектура, частые офисные ситуации. После него стандартный рабочий день — standup, обсуждение тикета, поход за код-ревью, «у меня сломалось» — собирается на знакомых блоках, а не переводится на лету.

## 2. Главное правило

<Callout type="purple" title="Ядро темы">
Tech English — это не «английский плюс термины», а **отдельный регистр** с устойчивыми коллокациями (`spin up a server`, `roll out a feature`, `push to prod`, `take down a service`), своими предлогами (`deploy **to** prod`, не «on»), своими ложными друзьями (`actual` ≠ актуальный) и набором готовых формул для процесса (`I'm blocked on…`, `Could you take a look at my PR?`, `LGTM`). Учить как речевые клише, а не как отдельные слова — иначе соберётся в дословный перевод и сразу прозвучит как русский в IT-русском.
</Callout>

## 3. Что входит в tech English

Условно — четыре области, в каждой свой словарь и свои клише.

| Область | Что туда входит | Когда нужно |
| --- | --- | --- |
| **Process** | agile, scrum, standup, sprint, backlog, retro, estimate, ticket, blocked | дейлики, планирование, оценки, статусы |
| **Code & Git** | repo, branch, commit, push, pull, merge, rebase, PR, review, conflict | работа в git, code review, обсуждение изменений |
| **Architecture** | frontend, backend, API, endpoint, database, cache, queue, microservice | дизайн, обсуждение системы, тех-собесы |
| **Useful phrases** | «I'm blocked on…», «Could you take a look at my PR?», «LGTM», «WFM» | чаты, код-ревью, переписка с командой |

Эти четыре области пересекаются, но словарь разный. На дейлике редко звучит `endpoint`, в архитектурной дискуссии — редко `retro`. Полезно держать их раздельно в голове: какой контекст — такой регистр.

## 8. Контрасты — близкие по смыслу пары

<Compare leftLabel="deploy / release / ship" rightLabel="Близкие, но не одно и то же">
  <div slot="left">
    <strong>deploy</strong> — техническое действие: положить артефакт на окружение.<br/>
    <em>We deploy to staging every commit.</em><br/><br/>
    <strong>release</strong> — событие: версия доступна пользователям/клиентам.<br/>
    <em>The 2.0 release is on Friday.</em><br/><br/>
    <strong>ship</strong> — выкатили в живое использование (разг.).<br/>
    <em>We shipped the new search.</em>
  </div>
  <div slot="right">
    Можно deploy без release (на staging — да, пользователи нет).<br/><br/>
    Можно release без deploy (бывает: артефакт собран, но не задеплоен).<br/><br/>
    <em>Ship</em> — почти всегда подразумевает «увидели пользователи». В разговоре чаще, чем <em>release</em>.
  </div>
</Compare>

<Compare leftLabel="bug / issue / ticket" rightLabel="Где какой регистр">
  <div slot="left">
    <strong>bug</strong> — конкретный сломанный кусок поведения.<br/>
    <em>There's a bug in the date picker.</em><br/><br/>
    <strong>issue</strong> — общее: проблема, предложение, вопрос. Шире, чем bug.<br/>
    <em>I opened an issue about the slow loading.</em><br/><br/>
    <strong>ticket</strong> — задача в трекере, любая.<br/>
    <em>I have three tickets this sprint.</em>
  </div>
  <div slot="right">
    Каждый bug — это issue. Не каждый issue — bug.<br/><br/>
    На GitHub «issue» — общий тип сущности (там же фичи, вопросы).<br/><br/>
    «Ticket» нейтрально про любую задачу в Jira / Linear. <em>Bug ticket</em> — задача про баг.
  </div>
</Compare>

<Compare leftLabel="client / server / frontend / backend" rightLabel="Где граница">
  <div slot="left">
    <strong>client</strong> — кто отправляет запрос.<br/>
    Браузер, мобильное приложение, скрипт.<br/><br/>
    <strong>server</strong> — кто принимает и отвечает.<br/>
    Web-сервер, API, бэкенд.<br/><br/>
    <strong>frontend</strong> — UI-часть.<br/>
    <strong>backend</strong> — серверная логика и данные.
  </div>
  <div slot="right">
    client/server — про <em>роли в коммуникации</em>.<br/><br/>
    frontend/backend — про <em>где живёт код</em> в продукте.<br/><br/>
    Frontend — всегда client-side. Backend — всегда server-side. Но client/server описывает шире — например, два бэкенд-сервиса между собой тоже client и server.
  </div>
</Compare>

<Compare leftLabel="prod / staging / dev" rightLabel="Окружения">
  <div slot="left">
    <strong>prod (production)</strong> — где живут пользователи. Сломаешь — все увидят.<br/><br/>
    <strong>staging</strong> — копия prod для финальной проверки. Те же данные структурно, без живых пользователей.<br/><br/>
    <strong>dev / develop / local</strong> — рабочее окружение разработчика или общая dev-среда.
  </div>
  <div slot="right">
    Поток: <em>local → dev → staging → prod</em>.<br/><br/>
    Деплой идёт <em>to</em>: <em>deploy to staging</em>, <em>push to prod</em>.<br/><br/>
    <em>On prod</em> уместно для статуса: <em>the bug only appears on prod</em>. <em>Deploy on prod</em> — не норма.
  </div>
</Compare>

<Compare leftLabel="merge / rebase / squash" rightLabel="Способы интегрировать ветку">
  <div slot="left">
    <strong>merge</strong> — создаёт merge-commit, история сохраняется.<br/>
    <em>Merge the feature branch into main.</em><br/><br/>
    <strong>rebase</strong> — переписывает коммиты ветки поверх main, линейная история.<br/>
    <em>Rebase your branch onto main.</em>

  <div slot="right">

    <em>Squash and merge.</em><br/><br/>

  </div>
</Compare>

## 9. Мнемоника и лайфхаки

<Callout type="teal" title="Артикль перед IT-аббревиатурами">
Артикль определяется **звуком первой буквы аббревиатуры, не написанием**. `MVP` читается как «эм-ви-пи» — начинается на гласный звук, значит **an MVP**. `SQL` чаще читается «эс-кью-эл», тоже на гласный — **an SQL query**. `API` — «эй-пи-ай», но первый звук [eɪ] — гласный, **an API**. А `UI` — «ю-ай», первый звук [j] — согласный, поэтому **a UI**. `URL` — «ю-ар-эл», тоже **a URL**. Запомни: проговори вслух первую букву, и артикль вылезет сам.
</Callout>

## 10. UK vs US — мелкие отличия

<Callout type="blue" title="Мелкая разница, на которую можно не сильно отвлекаться">
В tech-словаре UK/US почти не расходятся — индустрия глобальная, в чатах преобладает US-вариант. Но точечно: `whilst` (UK) против `while` (US, и в коде везде), `colour` (UK) против `color` (US, и в CSS), `licence` (UK noun) против `license` (US универсально, и в коде/лицензиях). Глагол `to license` — везде с `s`. В терминологии разработки крупных расхождений практически нет: `deploy`, `release`, `ship`, `PR`, `repo` — одинаково везде. Можно спокойно опираться на US-норму — её точно поймут, а UK-коллеги привыкли.
</Callout>

## 12. Типовые ошибки русскоязычных

| Неверно | Верно | Почему |
| --- | --- | --- |
| I have a problem with X | I'm running into an issue with X | «I have a problem» звучит тяжело и драматично. На работе — `running into an issue / hitting a snag / seeing a weird thing`. Легче по тону. |
| What is the actual status? | What is the current status? | `Actual` — это «реальный / фактический», не «актуальный». «Актуальный» = `current / up-to-date / relevant`. Один из самых частых ложных друзей. |
| Make me a code review, please | Could you review my PR? | `Make a code review` — калька с русского. По-английски: `review my PR`, `take a look at my PR`, `do a quick review`. Глагол — `review`, не `make`. |
| Please accept my PR | Please approve my PR | В git/GitHub PR не «принимают» — его `approve` (одобряют), затем `merge` (сливают). `Accept` подходит для предложения о работе или приглашения, не для PR. |
| an UI / a API / a SQL query | a UI / an API / an SQL query | Артикль по **первому звуку аббревиатуры**. UI — [juː], согласный → `a`. API — [eɪ], гласный → `an`. SQL — [es], гласный → `an`. |
| I will ship the file to you | I'll send you the file | `Ship` в IT = выкатить в прод. Файл — `send`, `share`, `attach`. Иначе звучит, что ты деплоишь файл. |
| Please make git pull request | Please open a pull request | `Pull request` — это сущность в GitHub/GitLab. Создают её через `open / create / submit a PR`. `Git pull` — другая команда, она про подтягивание изменений из remote. Разные вещи, не смешивать. |
| Deploy on prod | Deploy to prod | `Deploy to <environment>`. Предлог `to` — направление действия. `On prod` уместно для состояния: `the bug shows up on prod`. Для деплоя — только `to`. |
| Our microservices are better than their monolith | We use microservices; they use a monolith | Архитектурный стиль — не оценка качества. Монолит — нормальное решение для многих систем. Сравнивать `better / worse` — звучит наивно. Описывай через `we use / they use / for our case we chose`. |
| I will commit it on master | I'll commit to main / push to main | `Commit to <branch>`, не `on`. Современная норма ветки — `main`, не `master` (в большинстве новых репо). |
| I made the deployment of the new version | I deployed the new version | Английский предпочитает глагол существительному: `deploy`, не `make a deployment`. `Refactor`, не `make a refactoring`. Короче и естественнее. |
| principal of least privilege | principle of least privilege | `Principal` = главный / директор / основная сумма. `Principle` = принцип. Омофоны, но в код-ревью пишут именно `principle`. |
| The design compliments the UX | The design complements the UX | `Complement` = дополнять. `Compliment` = хвалить. В техническом письме почти всегда нужен `complement`. |

| **Actions** | debug, fix, refactor, optimize, profile, benchmark, mock, stub, spin up, tear down, scale up/down/out |
| **Phrases** | I'm blocked on…, Could you take a look at my PR?, LGTM, WFM, hit a wall, get a build to pass, push to prod |

| Ловушка | Запомни |
| --- | --- |
| Артикль перед аббрев. | По первому звуку: an MVP, an API, an SQL, **a** UI, **a** URL |
| deploy to / push to prod | Предлог `to`, не `on` |
| review my PR | Глагол `review`, не `make a review` |
| approve / merge PR | Не `accept` |
| current ≠ actual | `Current` = актуальный; `actual` = фактический |
| ship = выкатить | Файл — `send`, не `ship` |

| main, не master | Современная норма ветки по умолчанию |
